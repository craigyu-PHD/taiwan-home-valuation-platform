# 全台房屋即時估價平台：系統設計與交付說明

## 設計原則

- 資料透明：每個價格結果都要能追溯到來源、資料版本、案例數與更新時間。
- 估價中立：不為買方或賣方刻意調高或壓低估值。
- 結果可解釋：輸出價格區間、信心分數、案例列表、主要因素與限制。
- 低信心敢拒估：資料不足、特殊產權或非標準物件時不硬給具體價格。
- 手機端好用：搜尋、地圖選點、底部抽屜與主要 CTA 都適合單手操作。

## 前後端架構

- 前端：React + Vite + TypeScript + Leaflet。
- 地圖：Leaflet + OpenStreetMap 圖磚。正式站需保留 attribution，避免大量預抓圖磚。
- 地址解析：本地候選地名優先；必要時使用 Nominatim，需節流、快取、顯示來源。
- 資料同步：`scripts/sync-moi-data.mjs` 下載內政部本期 CSV ZIP，轉為標準交易 JSON。
- 正式後端建議：Node.js API + PostgreSQL/PostGIS。以排程同步 Open Data，地理定位後建立空間索引。

## 資料來源

- 主要來源：內政部不動產成交案件實際資訊 Open Data。
- 本期下載端點：`https://plvr.land.moi.gov.tw/Download?type=zip&fileName=lvr_landcsv.zip`
- 政府資料集：本期發布之不動產買賣實價登錄批次資料。
- 地圖與地理編碼：OpenStreetMap、Leaflet、Nominatim 或自架 Nominatim。
- 銀行估價：只整理公開網站來源，作為外部參考。

## 估價邏輯

1. 地址解析成縣市、行政區、路段、經緯度。
2. 依空間距離抓取 3.5 公里內、36 個月內成交案例。
3. 優先權重：同社區 > 同路段 > 同行政區 > 同縣市。
4. 加權條件：距離、交易時間、建物型態、坪數、屋齡、樓層、車位。
5. 計算加權 P15 / P50 / P85 單價，再乘以權狀坪數。
6. 依信心分數調整區間寬度。
7. 若特殊物件或案例不足，輸出低信心或不適合自動估價。

## 信心分數

提高信心：

- 同社區或同路段案例多。
- 近 6 到 12 個月成交多。
- 1 公里內案例多。
- 坪數、屋齡、樓層與型態相近。
- 使用者資料完整。

降低信心：

- 地址解析不明確。
- 案例數不足或單價離散度高。
- 屋況未知。
- 特殊物件或特殊產權。
- 非標準住宅類型。

## 銀行估價網站參考策略

可做：

- 在結果頁整理銀行公開估價或房貸試算網站。
- 明確標示銀行結果不是本平台估價，也不代表核貸結果。
- 保留本平台估價與銀行外部網站的來源差異。

不可做：

- 未授權批次爬取銀行結果。
- 繞過 CAPTCHA、簡訊驗證、登入或反自動化限制。
- 把銀行試算結果包裝為本平台自行估價。
- 移除銀行原始來源與限制說明。

## 頁面與流程

1. 首頁：方法摘要、地址搜尋、地圖入口、估價原則。
2. 地圖估價頁：搜尋、定位、點選地圖、周邊成交、篩選、底部抽屜。
3. 估價條件頁：坪數、樓層、屋齡、車位、屋況、使用狀態、特殊狀況。
4. 估價結果頁：價格區間、信心分數、案例、地圖、原因、提醒、銀行網站來源。
5. 區域行情頁：行政區行情摘要。
6. 方法與免責聲明頁：資料來源、免費 API 限制、模型限制與正式風險揭露文案。

## MVP 上線注意

- 第一版只支援住宅大樓、華廈、公寓、套房。
- 內政部 Open Data 地址多為去識別化或門牌格式不一，正式地圖距離估價需完成地理定位管線。
- Nominatim 不適合大量自動 geocoding；流量變大時應自架服務或改用合法資料批次處理。
- 銀行外部結果只能作比較參考，不應作為本平台主要估價來源。
